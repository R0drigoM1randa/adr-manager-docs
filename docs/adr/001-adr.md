## Estado
Aceptado

## Contexto y Problema
Actualmente nuestra plataforma utiliza una base de datos MySQL autogestionada en servidores locales. A medida que escalamos, nos enfrentamos a desafíos operativos significativos:
* Limitaciones de escalabilidad vertical (máximo de memoria y CPU alcanzado).
* Tiempo de inactividad durante las actualizaciones manuales del sistema.
* Altos costos operativos de mantenimiento, copias de seguridad y replicación.

Necesitamos una solución que permita alta disponibilidad, escalabilidad automática y menor esfuerzo de mantenimiento del equipo de infraestructura.

## Factores de Decisión (Decision Drivers)
* **Alta Disponibilidad:** Mínimo tiempo de inactividad esperado (99.99%).
* **Capacidad de Escalabilidad:** Debe soportar picos de tráfico estacionales sin intervención manual.
* **Tiempo de Implementación:** El equipo de desarrollo actual ya tiene experiencia con bases de datos relacionales (SQL).
* **Restricción Presupuestaria:** El costo total de propiedad (TCO) no debe superar el 15% del presupuesto actual de TI en los próximos 12 meses.

## Opciones Consideradas
* **Opción 1:** AWS Aurora PostgreSQL
* **Opción 2:** Amazon RDS for MySQL
* **Opción 3:** Continuar con servidores locales optimizados

## Resultado de la Decisión
Opción elegida: **Opción 1: AWS Aurora PostgreSQL**, porque ofrece la mejor arquitectura para el crecimiento a largo plazo y elimina por completo el mantenimiento de hardware, compensando el esfuerzo técnico de la migración de datos.

### Pros y Contras de las Opciones

### Opción 1: AWS Aurora PostgreSQL
* ¡Bueno, porque ofrece replicación rápida y escalado de lectura fluido.
* ¡Bueno, porque incluye mantenimiento automatizado y respaldos gestionados.
* ¡Mal, porque requiere migrar el esquema actual de MySQL a PostgreSQL.
* ¡Mal, porque implica una curva de aprendizaje para la optimización de queries específicos en PostgreSQL.

### Opción 2: Amazon RDS for MySQL
* ¡Bueno, porque tiene una curva de aprendizaje nula (mismo motor actual).
* ¡Bueno, porque la migración es transparente sin necesidad de cambiar el dialecto SQL.
* ¡Mal, porque el escalado es menos flexible en comparación con Aurora.
* ¡Mal, porque los costos operativos son mayores para clústeres multi-AZ.

### Opción 3: Continuar con servidores locales optimizados
* ¡Bueno, porque no genera costos adicionales de suscripción en la nube.
* ¡Mal, porque no soluciona el problema de fondo del cuello de botella en infraestructura.

## Consecuencias
* **Buenas:** Reducción del 40% en el tiempo de gestión de bases de datos por parte del equipo de DevOps. Capacidad de escalar hasta 5 réplicas de lectura de manera automatizada.
* **Malas:** Costo base ligeramente más alto que un RDS tradicional. Se requiere tiempo de desarrollo para refactorizar queries que utilizaban funciones propietarias de MySQL.
